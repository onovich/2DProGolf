# 2D Pro Golf 开发工作流

## 1. 目的

这份文档只规定一件事：功能开发必须以“可验证、可推送、可回滚”为单位推进，而不是长时间堆积在本地工作区。

## 2. 强制纪律

从现在开始，默认执行以下纪律：

1. 做完一个明确功能后，先做本地验证。
2. 本地验证通过后，立刻提交。
3. 提交完成后，立刻推送到 origin/main。
4. 推送后，立刻检查 GitHub Pages 线上结果是否已更新。
5. 只有这一轮线上验证结束后，才能开始下一个功能。

这里的“一个明确功能”指的是可单独描述、可单独验证、可单独回退的一段改动，例如：

1. 标题页接管启动流程。
2. 教学结业后进入世界地图。
3. 正式赛完赛后回到大地图并高亮下一站。

不允许再出现“已经连续写了很多功能，但大部分还没推”的情况。

## 3. 标准执行顺序

每次开发默认按这个顺序执行：

1. 明确本次只做哪个功能。
2. 只收集完成该功能所需的最小上下文。
3. 完成实现。
4. 运行最小必要验证。
5. 修掉本轮验证暴露的问题。
6. 再跑一次验证确认通过。
7. git add -A
8. git commit -m "..."
9. git push origin main
10. 打开并验证线上 Pages 效果。
11. 记录结果，再进入下一项。

## 4. 验证要求

至少满足以下两类中的一类，最好两类都做：

1. 构建验证：npm run build
2. 行为验证：本地浏览器或线上 Pages 直接确认目标流程

如果功能会影响上线效果，仅本地构建通过还不够，必须补一次线上 Pages 验证。

## 5. 提交要求

提交信息要直接描述用户能感知的改动，不要写成模糊的大杂烩。

推荐：

1. feat: route tutorial graduation to world map
2. fix: restore title-first startup flow
3. feat: return competition completion to the world map

避免：

1. update stuff
2. wip
3. misc changes

## 6. 例外条件

只有以下情况可以暂不立即推送：

1. 功能还没有形成可验证闭环。
2. 当前改动会明显破坏主流程，必须先补齐同一功能的收尾修复。
3. 远端或 Pages 服务本身不可用。

即使出现这些例外，也应以“尽快收敛到一次可推送状态”为目标，而不是无限期堆积本地改动。

## 7. 当前项目特别说明

本仓库的 Pages 部署由 .github/workflows/deploy.yml 负责，并且会在 main 分支 push 后自动触发。

因此，这个项目的默认开发闭环就是：

1. 本地实现。
2. 本地验证。
3. 提交。
4. 推送。
5. 验证线上。

这是默认流程，不再是可选项。
