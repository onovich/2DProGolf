# 2D Pro Golf RPG 化执行任务清单

## 0. 当前执行现状

这份 backlog 最初对应的是“从原型走向轻 RPG 化”的建设期。到当前版本为止，项目状态已经明显前移，所以需要先用这段现状更新来重新理解优先级。

当前已经完成的主线能力：

1. 教程 1 到 5 洞与序章引导完整可玩。
2. 城市据点、NPC 对话、正式赛事、世界地图和章节结算已经形成完整推进链。
3. 白昼冠城、世界冠军赛、通天塔终章，以及终章后的结尾页和通关总览页均已接入。
4. 轻量成长系统已经通过巡回心得落地，并作用于风、落地和推杆读线。
5. 通关总览页已经具备成绩单、阶段成绩、打法画像、章节证据和尾声文案。

当前更高优先级的工作重心：

1. 收尾表现层精修，而不是继续盲目扩赛区。
2. 手动验收与调试入口补齐，尤其是终章和通关页。
3. 持续做维护性重构，防止 gameEngine.js 的流转层重新退回硬编码分支。
4. 在不稀释高尔夫主玩法的前提下，继续补更强的可复盘反馈和可读性。

当前建议的 backlog 阅读方式：

1. 下面的 P0、P1、P2 章节主要用于回看历史方向，不应再按原样顺序机械执行。
2. 如果要开新任务，应优先围绕“终章验收、通关页体验、开发调试能力、中后期可维护性”来拆。
3. 若新增内容系统，需要同步回答它是否真的强化了击球、读风、读坡和落点判断。

## 1. 使用方式

这份文档用于把总策划拆成可执行任务。使用原则如下：

1. 先完成 P0，再进入 P1 和 P2。
2. 优先实现能反复复用的底层能力，不先堆一次性内容。
3. 任何任务都要回答一个问题：它是否强化了高尔夫主玩法的可读性、可学习性和长期推进动力。

## 2. 美术实现约束

项目当前美术方向确定为纯代码方案，执行约束如下：

1. 优先使用 Canvas 程序化绘制。
2. UI 图形优先使用 SVG 代码绘制。
3. 图标优先使用开源图标库或自绘简化 SVG 图标。
4. 不主动引入位图资源作为常规方案。
5. 只有在关键表现确实无法靠程序化图形和 SVG 达成时，才列为待补外部美术资源。

## 3. 里程碑总览

### 里程碑 A：原型稳定化

目标：把当前原型调成可持续迭代的基础版本。

验收标准：

1. 风力不再频繁抢夺玩家控制权。
2. 关键参数可配置。
3. 有固定测试球场和固定随机种子。
4. 原型难度曲线可稳定复现。

### 里程碑 B：序章可玩化

目标：玩家能从开场进入教学、完成第一段剧情并理解长期目标。

验收标准：

1. 有序章开场演出。
2. 有精灵向导。
3. 有第一座城市和基础赛事。
4. 有赛前、赛后、对话和结算界面。

### 里程碑 C：主循环成型

目标：玩家能在多个城市之间推进，体验差异化赛事和对手。

验收标准：

1. 至少 2 到 3 座城市完整可玩。
2. 有对手、名次和赛事规则变体。
3. 有基础成长和内容解锁。

## 4. 程序任务清单

### P0 程序任务

1. 重构参数结构。
说明：把风力、球杆、洞型、城市、赛事和教程参数从引擎硬编码中抽出。

2. 风力系统重做。
说明：下调总体风强，加入球杆受风系数、逆风最小推进保障和风偏预估。

3. 固定种子系统。
说明：支持固定随机种子，用于测试和内容复现。

4. 洞型模板系统。
说明：从完全自由生成改为模板加变体生成。

5. 调试面板。
说明：提供开发态调参面板，可切换风力、地形、洞型、球杆参数。

6. 比赛状态机基础重构。
说明：为序章、城镇、赛前、比赛中、赛后、剧情对话预留清晰状态切换。

7. AI 对手基础框架。
说明：先做简化版对手成绩与出杆模拟，不必一开始完全可视化逐杆演算。

8. 文本事件系统。
说明：支持简单对话节点、人物说话、选项和触发条件。

### P1 程序任务

1. 世界地图与选城流程。
2. 多赛事类型框架。
3. 成长系统基础数据结构。
4. 城市规则修正器。
5. 观众 NPC 交互触发。
6. 赛后结算与排名面板。

### P2 程序任务

1. 装备与部件系统。
2. 更复杂的 AI 策略切换。
3. 支线触发器和关系推进系统。
4. 终章专属演出逻辑。

## 5. 美术与界面任务清单

### P0 美术任务

1. 建立 UI 视觉规范。
说明：在延续当前深色玻璃风格的基础上，补充章节标题、对话框、结算卡片、城市选择卡和训练界面样式。

2. 建立 SVG 图形规范。
说明：统一风向图标、赛事图标、训练图标、城市特征徽记、对话气泡等图形语言。

3. 建立程序化角色表现方案。
说明：先不做复杂人物立绘，可用头像底板、剪影、徽章、职业图形和色块组合表达角色身份。

4. 建立城市主题规则。
说明：不同城市主要通过配色、地形纹理、粒子、旗帜造型、UI 点缀差异来区分，不依赖大批外部图像。

5. 建立球场装饰规范。
说明：树木、灯柱、看台、石块、旗帜、告示牌等都优先程序化绘制或 SVG 叠加。

### P1 美术任务

1. 对话立绘占位升级。
2. 城镇界面布局与地图节点界面。
3. 赛事转场动效。
4. 终章通天塔的抽象演出语言。

### P2 美术任务

1. 特殊角色专属徽记。
2. 冠军赛和神灵之所的高级视觉效果。
3. 如有必要，再补极少量外部资源需求清单。

## 8. 负责人视角的交付建议

如果由单人或极小团队推进，建议按以下交付节奏执行：

1. 每个阶段先做一个完整纵切片，不先追求系统齐全。
2. 每新增一个系统，就立刻绑定一个实际内容场景验证它。
3. 程序化内容和文本内容同步推进，避免后期只剩空壳系统。
4. 美术层只做“足够支持氛围和可读性”的程度，不做资源黑洞。

## 9. 当前立刻可开工事项

1. 做一轮终章到通关页的真实手动验收，并记录移动端问题。
2. 继续补通关页的复盘能力，例如更细的阶段证据与风格解释。
3. 视需要增加更多中段调试预设，减少内容验收成本。
4. 对 gameEngine.js 做阶段性维护性整理，优先收敛与 phase 流转直接相关的逻辑块。
5. 补 README 和现状文档，确保项目入口认知跟当前版本一致。
6. 在不破坏主玩法权重的前提下，再评估是否需要追加新的终章字幕或表现层。
7. 如果继续扩内容，优先做“现有系统上的薄增量”，不要先开重系统。
