# 2D Pro Golf 开发者接手指南

## 1. 先用一句话理解这个项目

这是一个以程序化 2D 高尔夫为主玩法、以轻 RPG 外壳做推进和复盘的单机项目，主线已经打通到通天塔终章与通关后的总结页。

如果你刚接手，不要先把它理解成“待完成的大型 RPG”，而应该理解成“已经有完整纵切版本、正在进入收尾和稳固阶段的高尔夫内容产品”。

## 2. 现在已经有什么

当前玩家可体验的完整链路包括：

1. 教程 1 到 5 洞。
2. 序章引导与精灵对话。
3. 城市据点与 NPC 对话。
4. 数据驱动正式赛事链。
5. 世界地图与赛区推进。
6. 章节结算与心得奖励。
7. 白昼冠城、世界冠军赛、通天塔终章。
8. 终章结尾页。
9. 通关总览页，包括成绩、阶段拆解、打法画像和章节证据。

从产品角度看，这已经是一条完整主线，而不是只有系统雏形。

## 3. 现在最重要的认知边界

### 3.1 什么是这个项目的核心

核心仍然是高尔夫击球体验：

1. 读风。
2. 读坡。
3. 读落点。
4. 选杆与节奏判断。

轻 RPG 层的职责是：

1. 给玩家长期目标。
2. 给每段推进更明确的情绪落点。
3. 让每次完赛更可复盘。

### 3.2 什么是不应该做的

1. 不要为了“更像 RPG”去加重装备、等级或数值膨胀。
2. 不要继续无节制扩主线城市，当前主线已经闭合。
3. 不要把新的流程判断重新散落回 gameEngine.js 的临时 if 分支。
4. 不要做会削弱高尔夫判断权重的自动化辅助或过强成长。

## 4. 先看哪里

### 4.1 先看文档

建议阅读顺序：

1. docs/development-workflow.md：先理解当前仓库的强制开发纪律，尤其是“每完成功能就提交、推送并验证 Pages”。
2. docs/current-project-status.md：快速理解当前版本现状。
3. docs/development-backlog.md：理解历史 backlog 和当前优先级如何变化。
4. docs/rpg-design-plan.md：理解最初的产品设计原则和长期方向。

### 4.2 再看代码

建议阅读顺序：

1. src/logic/engine/gameEngine.js：先理解 phase、state 和 runFlowAction(action)。
2. src/data/content/*.js：再理解教程、城市、赛事、章节结算、终章和心得是怎样数据化的。
3. src/view/screens/gameScreen.js：最后看各个面板对应的 DOM 结构。
4. src/view/styles/game.css：再看终章和通关页的表现层组织方式。

## 5. 当前最重要的结构资产

### 5.1 数据驱动内容层

当前最值得保护的是 src/data/content 这层。

它已经承载：

1. tutorial.js
2. city.js
3. competition.js
4. worldMap.js
5. chapterRecap.js
6. playerPerk.js
7. ending.js

新增内容时，优先想办法把新语义长进这里，而不是先写死在引擎里。

### 5.2 阶段驱动流转

当前高价值 phase 包括：

1. city
2. world-map
3. chapter-recap
4. ending
5. tour-summary

新的推进逻辑应该尽量通过 runFlowAction(action) 接入，而不是旁路修改多个面板显示状态。

## 6. 当前最大的技术风险

### 6.1 gameEngine.js 继续膨胀

它已经是主控中心，继续在这里叠一次性逻辑会很容易回退成硬编码泥团。

处理原则：

1. 每次新增功能后，顺手判断是否能收敛到已有 helper 或 phase 分支。
2. 优先做小规模维护性整理，而不是等到大爆炸式重构。

### 6.2 终章与通关页难以回归验证

这个问题已经部分解决，因为项目现已支持：

1. ?debug=ending
2. ?debug=tour-summary

也就是说，终章与通关页现在可以直接验，不需要重新打一遍主线。

## 7. 你接手后第一天最该做什么

1. 跑一次 npm run build，确认基础环境正常。
2. 用 ?debug=ending 和 ?debug=tour-summary 分别看一遍终章页和通关页。
3. 读 docs/current-project-status.md 和这个 handoff guide。
4. 再去看 src/logic/engine/gameEngine.js 的 phase 和 flow action 组织。

## 8. 当前最推荐的工作方向

如果你是接手后准备继续推进的人，最合适的切入顺序是：

1. 终章与通关页的手动验收和移动端优化。
2. 小规模维护性重构，继续保护 phase + flow action 结构。
3. 通关页与终章页的表现层精修。
4. 必要时补更多中段调试预设。
5. 最后再评估是否需要极少量新增内容。

## 9. 一句话结论

接手这个项目时，最重要的不是“继续往前铺更长的内容线”，而是先守住已经做出来的这条完整路线，让它更稳定、更可验收，也更像一个真正打磨过的版本。
